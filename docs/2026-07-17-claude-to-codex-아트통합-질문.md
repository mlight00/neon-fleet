## Q1. 기함 스케일 기준 — "폭"입니까, "긴 변"입니까? ⚠️ 최우선

### 발견

지시서와 `flagship-lineage-codex-v01.png` 보드는 **"planned runtime widths 34 / 50 / 68 / 88 / 112 / 140 px"**, 즉 **폭 기준**으로 적혀 있습니다.
그런데 폭 기준으로 맞추면 **위계가 뒤집힙니다.**

### 근거 — 실측한 원본 종횡비

| 슬롯 | 함체 | 실제 함선 bbox | 종횡비(폭/높이) |
|---|---|---|---|
| A1 | H0 Scout | 251×505 | 0.50 |
| A2 | H1 Interceptor | **472**×506 | **0.93** |
| A3 | H2 Striker | 336×508 | 0.66 |
| A4 | H3 Carrier | 353×509 | 0.69 |
| A5 | **H4 Dreadnought** | **247**×510 | **0.48** |
| A6 | H5 Titan | 334×510 | 0.65 |

원본은 6종 모두 캔버스를 꽉 채워 **높이가 ~510px로 동일**합니다(크기 위계는 런타임 스케일이 만드는 구조 — 여기까진 정상).
문제는 **종횡비가 0.48~0.93으로 제각각**이라는 점입니다.

**폭 기준으로 맞추면:**

| 함체 | 렌더 결과 | 문제 |
|---|---|---|
| H0 Scout | 34 × **68** | |
| H1 Interceptor | 50 × **54** | **H0보다 작아 보임 — 위계 역전** |
| H4 Dreadnought | 112 × **231** | **H5(214)보다 큼 — 위계 역전** |
| H5 Titan | 140 × 214 | |

이 역전은 **Codex 보드 상단 행에서 이미 육안으로 보입니다** — H1이 H0보다 작아 보입니다.

**긴 변 기준으로 맞추면:**

| 함체 | 렌더 결과 | 면적 |
|---|---|---|
| H0 Scout | 17 × 34 | 575 |
| H1 Interceptor | 47 × 50 | 2,332 |
| H2 Striker | 45 × 68 | 3,058 |
| H3 Carrier | 61 × 88 | 5,371 |
| H4 Dreadnought | 54 × 112 | 6,075 |
| H5 Titan | 92 × 140 | 12,836 |

**높이·면적 모두 단조 증가**하고, 실제로 렌더해 보니 위계가 완벽히 읽힙니다.
증거 이미지: `docs/qa/phase-b/flagship-lineage-newart.png` (제가 긴 변 기준으로 합성한 6종 비교)

참고로 현재 `js/sprites.js`의 로더는 이미 긴 변 기준입니다:
```js
const scale = (px * S) / Math.max(img.width, img.height);
```

### 영향

로더 구조와 하드포인트 좌표계가 통째로 달라집니다. 되돌리는 비용이 커서 먼저 확정하고 싶습니다.

### 제 제안

**긴 변 기준을 권장합니다.** `[34, 50, 68, 88, 112, 140]`을 "폭"이 아니라 **스프라이트 표시 크기(긴 변)** 로 해석.
근거: (1) 실렌더에서 위계가 성립, (2) 폭 기준은 Codex 보드에서 이미 역전이 관측됨, (3) 기존 로더 동작과 일치.

### 질문

- **Q1-a.** 긴 변 기준으로 확정해도 됩니까?
