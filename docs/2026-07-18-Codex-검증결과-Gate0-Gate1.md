## 3. Gate 0 검증

### 3.1 통과 항목

1. **배경 이동 방향**
   - `backdropTileY`와 `backdropLayerY`가 스크롤 증가 시 화면 y를 증가시키므로 배경과 입자가 위에서 아래로 흐른다.
   - 관련 자동 테스트가 방향 부호와 속도 순서를 확인한다.

2. **섹터별 배경 6종**
   - S1~S6가 서로 다른 파일로 1:1 연결됐다.
   - 자산 랩에서 여섯 배경의 색·랜드마크·실루엣 차이를 확인했다.

3. **일반 적 12종·캠페인 보스 6종 교체**
   - 자산 랩이 `자산 24종 + 보스 페이즈 8컷 렌더 완료 — 로드 실패 0`을 표시했다.
   - 로컬 HTTP 검사에서도 24개 모두 200이었다.

4. **B22/B7 단일 몸체**
   - B22는 모든 주요 단계에서 연결된 기본 몸체를 유지하고 손상은 VFX로 표현한다.
   - B7은 1~3단계에서 단일 몸체를 유지하고 4단계만 탈출 코어를 사용한다.
   - 기존의 좌표가 다른 구형 부품을 새 몸체 위에 다시 합성하지 않는다.

### 3.2 남은 증거 보완

작업 지시서는 S1~S6 각각을 실제 게임에서 5초 이상 재생한 1280×720 캡처 6개와 390×844 캡처 6개를 요구한다. 제출된 `docs/qa/gate0`에는 자산 보드·보스 단계 보드 중심의 3개 파일만 있다.

Gate 0 기능은 통과로 판정하지만, Claude는 재제출 때 실제 게임 캡처 12개와 각 캡처의 섹터·해상도·재생 시간을 목록으로 추가해야 한다.

## 4. Gate 1 차단 결함

### G1-01. 플레이어 조작을 매 프레임 자동 운전이 덮어쓴다 — 차단

`coreLoopUpdate()`가 조건 없이 `coreLoopAutopilot()`을 호출하고, 자동 운전은 `input.targetX`를 매 프레임 다시 쓴다. 따라서 `?coreLoopTest=1`에서 플레이어가 좌우 이동을 해도 다음 프레임에 자동 조향이 우선한다.

Gate 1은 플레이어가 8분 동안 선택하고 회피하는 수직 슬라이스여야 한다. 현재 구현은 플레이가 아니라 측정용 자동 재생이다.

**수정 조건**

- `?coreLoopTest=1`: 사람 플레이 전용, 자동 운전 금지
- 별도 `?coreLoopMeasure=1` 또는 명시적 옵션: 자동 측정 전용
- 화면과 로그에 `playable`/`measurement` 모드를 구분해 표시

### G1-02. 시작 조건과 결과 기록이 사실과 다르다 — 차단

사람이 접속하는 유일한 Gate 1 진입점도 다음 값을 강제로 적용한다.

- `surv.hullMax = 1400`
- `sq.count = 70`
- `sq.banked = 260`
- `sq.tier = 2`

그런데 결과 화면에는 `startTier: 0`을 하드코딩한다. `startHull`은 1,400으로 바꾸기 전에 저장한 100이므로 시작 내구도 역시 실제 시작 상태와 다르게 보고될 수 있다.

구현 결과서의 “실제 플레이 내구도는 100”이라는 설명도 사실이 아니다. 공개 캠페인은 아직 구형 드론 체력 모델이고, Gate 1 플레이 진입점은 1,400을 쓴다. 현재 코드에는 사람이 플레이하는 내구도 100 모드가 없다.

또한 2:15 승급은 tier 2에서 tier 3으로 진행되므로 H0→H1 사건이 아니다.

**수정 조건**

- 실제 플레이는 H0, 무기 1개, 소규모 함대, 내구도 100에서 시작
- 측정 모드 보정값은 플레이 모드와 완전히 분리
- 결과 화면은 실제 시작 스냅샷을 사용하고 티어를 하드코딩하지 않음

### G1-03. 선택이 없고, 행동 변화 지표에 실제 변화가 없는 사건도 기록된다 — 차단

출격 시 빌드는 `railStorm`으로 고정된다. 이후 두 번째 무기, 무기 강화, 공명, 프레임 모두 시간표가 자동 적용한다. 브라우저의 플레이 DOM에도 사운드·일시정지·차지 버튼 외 선택 UI가 없었다.

행동 변화 사건은 40초마다 자동 발생하지만 무기 레벨은 3에서 막힌다. 레벨이 이미 최대여도 `metrics.choice(..., {behavior:true})`는 계속 기록한다. 따라서 제출된 `[30,70,110,…]`은 실제 보이는 행동 변화가 아니라 스케줄러 사건 시각이다.

`weapon-loadout.js`의 2슬롯 로직도 런타임에서 쓰지 않고, 런타임은 `sq.weapon`과 `sq.wing`을 별도로 조작한다.

**수정 조건**

- 0:00 시작 무기 선택
- 0:30 전후 실제 행동 변화 카드 선택
- 1:15 두 번째 무기 획득 시 추가/교체 확인
- 5:30 프레임 선택
- 레벨 상한에서 아무 변화가 없으면 행동 변화로 기록하지 않음
- 선택 직후 발사 모양·발사 수·속도·범위 중 하나 이상이 눈에 띄게 달라짐

### G1-04. 긴급 재건·수리·보호막 기능이 실제 런타임에 연결되지 않았다 — 차단

`survivability.js`에는 `repair`, `canEmergencyRebuild`, `doEmergencyRebuild`, `addShield`가 있으나 다른 게임 코드에서 호출되지 않는다. 자동 테스트는 독립 모듈만 검사한다.

구현 결과서의 “긴급 재건 출격당 1회”는 런타임 구현 사실이 아니다. 최대 횟수 데이터와 유닛 테스트만 존재한다.

**수정 조건**

- 실제 피해·선택·결과 로그에 긴급 재건 1회가 연결됨
- 비용, 순양함 복구 수, 발동 연출, 발동 시각을 기록
- 수리·보호막도 획득 경로와 기회비용을 가진 실제 선택으로 연결
- H0·내구도 100 사람 플레이 생존 곡선을 별도 측정

### G1-05. 지휘 프레임 3종 중 2종의 자동 행동이 완성되지 않았다 — 차단

- 어썰트: 처치 누적으로 피해 배수가 연결되어 있다.
- 캐리어: `volleyMult`를 계산하지만 런타임에서는 원형 VFX만 그린다. 추가 사격이나 피해가 없다.
- 페이즈: FLOW가 100에 도달하면 기존 FLOW 로직이 즉시 RUSH를 시작하고 FLOW를 0으로 되돌린다. 프레임 검사는 이 이후의 값을 읽으므로 `flow >= 100` 조건을 실제 플레이에서 만날 수 없다.

유닛 테스트는 `tickFrame(..., {flow:100})`을 직접 호출하므로 통과하지만 실제 통합 동작을 증명하지 않는다.

**수정 조건**

- 캐리어 동기화 사격이 실제 투사체 또는 명확한 피해를 발생
- 페이즈 자동 스킬을 FLOW/RUSH 전환 사건에 직접 연결
- 세 프레임 모두 실제 런타임 통합 테스트와 캡처 제출

### G1-06. 시커 빔이 표식 대상을 우선 추적하지 않는다 — 차단

레이저 명중 시 `reson.markId`에 적 객체를 저장한다. 그러나 `HomingMissile.pickTarget(world)`은 표식이나 `world.reson`을 전혀 확인하지 않고 기존 점수로 표적을 고른다.

현재는 유도 미사일이 우연히 표식 적을 맞히면 그 피해를 `seekerBeam`으로 분류할 뿐이다. “레이저 표식→미사일 우선 추적”이라는 행동 변화가 구현되지 않았다.

**수정 조건**

- 살아 있는 표식 대상이 있으면 일반 유도 미사일이 그 대상을 최우선 선택
- 표식 대상 파괴·만료 시 다음 표식으로 이동
- 표식 유무에 따른 미사일 경로 차이를 통합 테스트와 영상/연속 캡처로 증명

### G1-07. 피해·공명 기여도 측정이 실제 적용 피해가 아니다 — 높음

충돌 코드가 `hitByBullet()`을 호출한 뒤 원래 발사체 피해값을 그대로 `tallyBulletDamage()`에 넘긴다. 이 값은 다음을 정확히 반영하지 못할 수 있다.

- 남은 HP보다 큰 초과 피해
- 보스 단계·보호·BREAK 보정
- 실제로 흡수되거나 무효화된 피해

따라서 제출된 무기별 피해와 공명 기여도는 런타임의 실제 HP 감소량과 다를 수 있다.

**수정 조건**

- 충돌 전후 HP 차이 또는 `hitByBullet()` 반환값으로 실제 적용 피해를 집계
- 초과 피해·무효 피해·보호막 흡수 규칙 명시
- 같은 시드에서 합계 피해와 실제 감소 HP의 오차 검증

### G1-08. 보스 통과 수치를 충족하지 못했다 — 차단

- 레일 스톰 공명 기여도 4.7%는 통과선 8~30% 미만이다.
- 레일 스톰 B22 TTK 57~72초 중 일부는 상한 60초를 넘는다.
- B7 보스 랩 TTK 60~90초는 작업 지시서의 Gate 1 필수 측정이지만 제출되지 않았다.
- B22가 섹터 1~4 보스보다 강하다는 비교 측정이 없다.
- B22 HP를 현재 함대 화력에 비례시켜 강한 빌드의 성장을 상당 부분 상쇄하므로, 업그레이드로 강해졌다는 체감도 함께 확인해야 한다.

**수정 조건**

- 고정 시드와 동일 조건으로 B8~B11/B22 상대 비교
- B22 세 빌드 모두 45~60초
- B7 보스 랩 60~90초 및 부위 선택·약점 이동 증거
- 세 공명 모두 실제 적용 피해 기준 8~30%

### G1-09. 390×844에서 UI와 함대가 겹치거나 잘린다 — 높음

직접 실행한 390×844 화면에서 다음을 확인했다.

- 상단 오른쪽 사운드·일시정지 버튼이 무기·공명 HUD 글자와 겹침
- 70기 함대가 화면 왼쪽 밖으로 잘림
- 큰 함대가 하단 가장자리와 가까워 전투 가독성이 낮음

문서의 가로 스크롤 0은 사실이지만, 캔버스 내부 요소의 겹침·잘림까지 해결됐다는 뜻은 아니다.

**수정 조건**

- 390×844 안전 영역을 별도로 정의
- 상단 HUD와 버튼을 서로 침범하지 않게 배치
- 최대 함대 폭을 화면 안에 유지하거나 모바일 전용 압축 배치 적용
- B22/B7 약점과 안전 통로도 같은 해상도에서 검증

### G1-10. 제출된 브라우저 증거가 설명과 다르다 — 높음

- `docs/qa/gate1/coreloop-hud.png`의 실제 크기는 **480×800**이며 제출서가 말한 1280×720이 아니다.
- 모바일 캡처는 390×844지만 겹침·잘림이 없다는 판정은 이번 독립 실행 결과와 다르다.
- 8분 결과 화면의 전 항목을 확인했다는 설명에 대응하는 결과 화면 캡처가 없다.

**수정 조건**

- 파일 자체 픽셀 크기가 1280×720, 390×844인 캡처 제출
- 시작·두 번째 무기·첫 공명·H1·프레임·B22·결과 화면을 두 해상도에서 확인
- 캡처 목록에 실제 뷰포트, 게임 시각, 빌드, 시드 기록

## 5. 자동 테스트가 결함을 놓친 이유

363개 통과 자체는 유효하다. 다만 Gate 1 핵심 테스트 상당수가 다음 유형이다.

- 소스 문자열에 특정 함수명이나 필드가 있는지 확인
- 실제 게임과 분리된 순수 모듈을 직접 호출
- 통합 런타임에서는 도달하지 않는 값을 테스트에서 직접 주입

예를 들어 다음은 테스트를 통과하면서 실제 게임에서는 실패한다.

- 캐리어 프레임: `tickFrame`이 `volley`를 반환하는지만 검사하고 실제 사격은 검사하지 않음
- 페이즈 프레임: FLOW 100을 직접 전달해 통과하지만 실제 FLOW는 100에서 즉시 0이 됨
- 긴급 재건: 모듈 함수만 검사하고 게임 호출 경로가 없음
- 시커 빔: 표식 상태만 검사하고 유도 미사일의 표적 선택은 검사하지 않음
- Gate 1 배선: 실제 플레이 대신 소스 문자열 포함 여부를 다수 검사

재검증 때는 최소 한 개의 실제 런타임 통합 테스트가 각 핵심 기능을 끝까지 통과해야 한다.
