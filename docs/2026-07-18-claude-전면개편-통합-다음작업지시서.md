## 1. 최종 제품 목표

### 1.2 반드시 지켜야 할 다섯 가지 약속

1. 플레이어는 평균 30~50초마다 수치가 아니라 행동이 바뀌는 선택을 체감한다.

### 4.3 배경 진행 방향

- 플레이어 함대는 화면 아래에 있고 위쪽으로 전진하는 설정을 유지한다.
- 배경 구조물·입자·잔해는 화면 위에서 아래로 흘러야 한다.
- `scroll`이 증가할 때 그려지는 y좌표도 증가하는 하나의 규칙을 모든 배경 레이어에 적용한다.
- FAR < MID < NEAR 순으로 속도를 높인다.
- 신규 배경을 위아래 반전 복제하지 않는다. 8~12% 겹침 또는 가장자리 마스크를 사용해 이음매를 숨긴다.
- 배경 이미지 로드 실패 시에만 기존 절차형 배경을 폴백으로 사용한다.

### 4.4 섹터 1:1 매핑

`zoneForSector()`와 배경 생성 로직은 `(sector - 1) / 2` 방식으로 두 섹터를 묶지 않는다.

```js
const index = Math.min(5, Math.max(1, sector) - 1);
```

- 섹터 1~6 → S1~S6
- 섹터 7 이상 → 후속 전용 자산이 생기기 전까지 S6 유지
- 지역 이름, 색, 모티프, 배경 이미지 인덱스는 같은 기준 함수를 사용한다.

### 4.5 적과 보스 교체

- 캠페인 보스: B8, B9, B10, B11, B22, B7을 신규 WebP로 연결한다.
- 기존 충돌 크기와 공격 패턴은 Gate 0에서 유지한다.

### 4.6 B22 분리 현상 수정

### 5.2 런 디렉터

새 모듈을 분리해 시간 기반 사건을 관리한다. 이름은 프로젝트 구조에 맞춰 정하되, 한 파일에 UI·적 생성·세이브·렌더를 모두 넣지 않는다.

필수 책임:

- 경과 시간과 현재 Gate 1 사건 상태
- 다음 행동 변화 선택까지의 시간
- 무기 슬롯 해금 시점
- 공명 예고와 완성 가능 여부
- 함체 승급 제안 시점
- 정예 웨이브와 보스 시작 시점
- 결과 요약용 런 통계
- 일시정지·선택 화면 동안 시간 정지

개발 중에는 `?coreLoopTest=1`처럼 명시적인 테스트 진입점을 제공한다. Gate 1 통과 전 기존 공개 캠페인을 제거하지 않는다.

### 5.3 진짜 다중 무기 슬롯

현재 `Squad.weapon` 하나를 바꾸는 구조를 다음처럼 확장한다.

```js
weaponSlots: [
  { slot: 'main', weaponId: 'vulcan', level: 1, evolutionId: null, superId: null },
  { slot: 'wing', weaponId: 'laser', level: 1, evolutionId: null, superId: null }
]
```

세부 계약:

- Gate 1에서는 main·wing 2슬롯을 사용한다.
- 두 슬롯은 독립 발사 주기, 레벨, 진화, 피해 통계를 가진다.
- 새 무기 획득은 기존 무기 교체가 아니라 빈 슬롯 장착이 기본이다.
- 슬롯이 가득 찬 뒤에만 “강화” 또는 명시적인 “교체”를 제안한다.
- 무기 게이트를 지나가다 실수로 전체 빌드가 바뀌지 않게 선택 확인 규칙을 둔다.
- 기존 세 무기 발칸·레이저·유도 미사일과 이미 제작된 마운트·발사체 이미지를 재사용한다.
- 기함의 서로 다른 하드포인트에 두 무기 마운트가 동시에 보인다.
- HUD에 두 무기의 아이콘, 레벨, 진화 상태, 공명 진행을 함께 표시한다.

호환성:

- 기존 `weapon`, `weaponLv`, `weaponEvolutions` 기반 로직을 한 번에 삭제하지 않는다.
- 세이브·테스트·기존 캠페인용 호환 어댑터를 두고 새 런부터 슬롯 구조를 사용한다.
- 기존 단일 무기 테스트가 의미 없이 모두 깨지지 않게 마이그레이션 단계를 둔다.

### 5.4 무기 조합 공명 3종

`keystones.js`의 `lance_echo`는 이름에 “공명”이 들어가지만 무기 두 개의 조합 공명이 아니다. 이를 R6 완료 증거로 사용하면 안 된다.

| 조합 | 공명 이름 | 행동 변화 |
|---|---|---|
| 발칸 + 레이저 | 레일 스톰 | 발칸 명중 누적이 차면 레이저 하드포인트가 굵은 관통 레일 사격을 만들고, 조준선과 충격파가 함께 보임 |
| 발칸 + 유도 미사일 | 마이크로 미사일 포화 | 발칸 연속 명중 또는 처치가 소형 미사일 묶음을 생성하고 여러 표적으로 분산 추적 |
| 레이저 + 유도 미사일 | 시커 빔 | 레이저가 표적을 지정하면 미사일이 표식을 우선 추적하고, 표적 파괴 시 다음 적으로 표식이 이동 |

공통 계약:

- 피해 배수만 올리는 패시브 금지.
- 발동 조건, 화면 모양, 소리, 표적 선택 중 최소 두 가지가 바뀌어야 한다.
- 첫 공명은 4분 15초~4분 45초 사이에 확정적으로 완성할 수 있다.
- 공명 직전 20~40초 동안 HUD에 필요한 조합과 다음 변화의 짧은 예고를 표시한다.
- 공명 피해는 원래 두 무기의 통계와 구분해 집계한다.
- 공명이 공명을 재귀 호출해 무한 연쇄하지 않도록 공격 ID와 발동 잠금을 둔다.

### 5.5 강화 카드의 질

- 카드의 70% 이상은 선택 직후 화면에서 행동 변화가 보여야 한다.
- 피해 +N%, 연사 +N%처럼 숫자만 바뀌는 카드는 전체의 20%를 넘기지 않는다.
- 숫자 카드는 행동 변화 카드 사이의 보조 선택으로만 사용한다.
- 같은 무기에서 발사체 수, 궤적, 폭, 관통, 추적, 폭발, 표식, 연쇄 중 하나가 달라져야 한다.
- 선택 직후 0.8~1.5초의 강화 연출을 사용하되, 플레이어가 위험한 탄막 속에서 강제로 피격되지 않도록 짧은 무적 또는 탄 정리를 적용한다.

### 5.6 기함 내구도와 함대 자원 분리

현재 드론과 순양함이 사실상 기함 체력 역할까지 하므로 회수 드론이 많을 때 난이도가 무너진다. 다음 네 자원을 분리한다.

| 자원 | 역할 | 회복 규칙 |
|---|---|---|
| 기함 내구도 `hullIntegrity` | 실제 생존 체력 | 제한된 수리·지역 보상만 허용. 드론 회수로 회복 금지 |
| 드론 | 공격 밀도·순양함 건조·일부 카드 비용 | 적·보급에서 회수 가능 |
| 순양함 HP | 개별 호위함 생존 | 전투·노드가 바뀌어도 현재 HP 유지 |
| 보호막 | 명시적인 일회성 방어 | 카드·프레임 효과로만 생성 |

필수 피해 순서:

1. 명시적 보호막이 있으면 보호막 소비
2. 적탄이 순양함 실제 히트박스를 맞히면 해당 순양함 HP 감소
3. 기함 핵을 맞히면 기함 내구도 감소
4. 드론 수는 기함 내구도를 대신 지불하지 않음
5. 기함 내구도 0이면 드론·순양함 수와 무관하게 패배

추가 규칙:

- `js/main.js`의 노드 시작 시 `cruiserHp = []`로 만피 초기화하는 동작을 제거한다.
- 순양함 건조 시에만 새 개체가 만피로 추가된다.
- 파괴된 순양함은 회수 드론만으로 즉시 무한 재생성되지 않는다.
- 긴급 재건은 한 출격에 최대 1회이며, 명확한 비용·연출·로그를 가진다.
- H0~H5 승급은 내구도 최대치를 올릴 수 있지만 현재 체력을 매번 만피로 만들지 않는다. 회복량은 최대치 증가분 일부 등 제한 규칙을 사용한다.
- 수리는 선택 비용과 기회비용을 가진다. 자동 완전 회복 금지.
- 기함 시각 크기와 피격 핵 크기를 계속 분리한다.

권장 초기값은 테스트를 위한 시작점일 뿐 고정 답이 아니다.

- 기함 내구도: 100
- 일반 적탄: 6~10
- 정예·보스 주요탄: 12~20
- 충돌·레이저 경고 실패: 18~28
- 제한 수리: 최대 내구도의 15~25%

### 5.7 지휘 프레임과 FLOW/RUSH

기존 교리 세 가지는 새 지휘 프레임 세 가지로 흡수한다.

| 프레임 | 정체성 | 자동 스킬 예시 |
|---|---|---|
| Assault | 직접 화력과 정면 돌파 | 일정 처치마다 전방 화력 집중 |
| Carrier | 드론·순양함·미사일 운용 | 일정 주기 호위 공격 동기화 |
| Phase | 이동·탄막 회피·표식 | FLOW 최대 시 짧은 위상 돌파 |

- 기존 교리 효과를 중복 적용하지 않는다.
- FLOW/RUSH는 선택한 프레임의 자동 스킬과 연결해 의미를 설명한다.
- 프레임 선택은 단순 피해 배수 하나가 아니라 자동 행동, HUD 아이콘, 함체 발광 색, 사운드를 바꾼다.
- Gate 1에서는 세 프레임의 데이터 구조와 최소 한 번의 자동 발동을 검증한다.

### 5.8 B22와 B7 전투 보완

#### B22 네온 아비터

- 섹터 1~4 보스보다 실제로 강해야 한다.
- 목표 TTK: 의도한 Gate 1 완성 빌드 기준 45~60초.

- 단계 1: 고리 안전 통로와 좌우 무장 읽기.
- 단계 2: 한쪽 무장을 먼저 무력화할지, 중앙 STAGGER를 노릴지 선택.

- BREAK는 플레이어가 의도적으로 만들 수 있고, 성공하면 공격 중지와 25% 취약 시간이 명확히 보인다.
- 안전 통로 폭은 모바일 피격 핵 기준으로 실제 통과 가능해야 한다.

#### B7 하이브 퀸

- 항상 단일 개체다. 보스 복제 3기 금지.
- 목표 TTK: 25분 캠페인 완성 빌드 기준 60~90초. Gate 1 보스 랩에서도 같은 패턴 구조를 측정한다.
- 1단계: 좌우 알주머니가 잡몹과 탄막의 근원임을 학습.
- 2단계: 알주머니 중 하나를 먼저 파괴해 패턴 일부를 끊는 선택.
- 3단계: 왕관 약점 노출과 부채꼴 안전 지대 이동.
- 4단계: 심장 노출, 이전에 남긴 부위에 따라 최종 패턴이 달라짐.
- “총알을 많이 뿌리기” 대신 약점 위치, 부위 우선순위, 이동 경로, 공격 타이밍을 요구한다.
- 화면 밖 또는 UI 아래에 약점이 숨지 않게 390×844에서 검증한다.

## 7. Gate 2 — 25분 6개 지역 캠페인 계약

Gate 1이 승인되면 현재 노드형 캠페인을 아래 시간 기반 6개 위협 지역으로 재구성한다. 기존 맵 코드는 삭제하지 말고 마이그레이션이 끝날 때까지 폴백·개발 모드로 보존한다.

### 7.1 시간 구조

| 시간 | 지역·성장 | 핵심 사건 |
|---:|---|---|
| 0~1분 | 출격 | 시작 무기와 짧은 조작 학습 |
| 1~4분 | 섹터 1 COLD WAKE | 두 번째 무기, 첫 진화, H1, B8 결전 |
| 4~8분 | 섹터 2 PRISM GRAVE | 첫 공명, H2, B9 결전 |
| 8~12분 | 섹터 3 FURNACE LINE | 지휘 프레임 자동 스킬, H3, 세 번째 슬롯 예고, B10 결전 |
| 12~16분 | 섹터 4 BROKEN ARMADA | 함대 시스템 해금, 두 번째 공명, 편대 확장, B11 결전 |
| 16~20분 | 섹터 5 CHOIR VEIL | H4, 최종 무기 진화, B22 결전 |
| 20~23분 | 섹터 6 CROWN CORE | H5 타이탄, Apex, 최종 준비 |
| 23~25분 | CROWN NEST | B7 하이브 퀸 최종 결전과 엔딩 |

- 각 지역 보스의 등장 연출과 전투를 해당 시간 범위 안에 포함한다.
- B8~B11은 30~45초, B22는 45~60초, B7은 60~90초를 목표로 한다.
- B7 처치와 캠페인 엔딩이 25분의 명확한 완결점이다.
- 엔딩 뒤에는 원하는 플레이어만 `Overdrive`를 선택해 계속할 수 있다. 첫 클리어 보상이나 핵심 설계도를 받기 위해 Overdrive를 강요하지 않는다.

### 7.2 세 번째 슬롯

- Gate 2에서 `fleet` 슬롯을 해금한다.
- 세 번째 슬롯은 단순한 주포 복제보다 전투기, 기뢰, 지원 빔, 방어 드론 같은 함대 시스템을 우선한다.
- 12~16분 구간에 등장해 이미 완성된 두 무기 조합에 새로운 운용을 더한다.
- 세 슬롯 모두 화면과 결과 통계에서 구분한다.

### 7.3 함체 T0~T5 기능

각 승급은 외형만 바꾸지 않는다.

| 함체 | 기능 변화 예시 |
|---|---|
| T0 정찰정 | 기본 주포와 소형 드론 |
| T1 인터셉터 | 두 번째 하드포인트 또는 이동 반응 개선 |
| T2 스트라이커 | 공명 회로와 첫 지휘 프레임 |
| T3 캐리어 | 함대 시스템 슬롯과 전투기 편대 |
| T4 드레드노트 | 대형 측면 포대와 프레임 자동 스킬 강화 |
| T5 타이탄 | Apex 발동, 화면을 지배하는 최종 실루엣 |

### 7.4 경로 선택

- 전투 중에는 좌·우 레인 선택을 계속 사용한다.
- 약 4분마다 큰 경로 선택을 한 번 제공한다.
- 큰 선택은 위험/보상, 다음 적 구성, 수리 기회, 특정 무기 강화 확률 중 최소 두 요소를 바꾼다.
- 선택 때문에 플레이가 지나치게 자주 멈추지 않게 한다.
- 같은 선택 문구인데 결과가 사실상 같은 가짜 분기 금지.

### 7.5 적 구성 원칙

- 일반 소형 적 처치: 0.2~0.8초
- 중형 적 처치: 0.8~1.5초
- 정예 적 처치: 4~8초
- 지역 보스: 30~45초
- B22: 45~60초
- B7: 60~90초

적은 HP 벽이 아니라 빌드 시험지다.

- 군집 적: 광역 성능 시험
- 장갑·프리즘 적: 관통·측면 이동 시험
- 저격·포탑: 우선 표적 선택 시험
- 실드·보호자: 표적 순서 시험
- 스캐빈저: 보상 추격 위험 시험
- 감염 게이트: 레인 선택 시험
- 보스 부위: 단일 화력과 패턴 학습 시험

## 8. Gate 3 — 오늘도 내일도 다시 할 이유

### 8.1 영구 성장은 선택지를 연다

- 영구 공격력 총합은 캠페인 체감 기준 약 10~15% 이내로 제한한다.
- 메타 성장은 새 시작 무기, 새 프레임, 새 함대 시스템, 새 공명 변형, 새 위협 규칙을 연다.
- 오래 한 플레이어만 숫자로 압도하는 구조를 피한다.

### 8.2 첫 8~10회 출격 보장

각 출격이 끝날 때 최소 하나를 확정 공개한다.

- 새 무기 진화
- 새 공명 변형
- 새 지휘 프레임
- 새 적 도감
- 새 보스 기억
- 새 함체 장식 또는 엔진 색
- 새 위협 규칙

무작위 실패 때문에 아무것도 발견하지 못한 판이 연속으로 나오지 않게 pity 카운터를 둔다.

### 8.3 위협 단계 1~5

- 단순 HP·공격력 증가만 사용하지 않는다.
- 새 적 조합, 보스 패턴, 경로 규칙, 수리 제한, 선택 카드 조건을 단계별로 연다.
- 각 위협 단계는 “이번에 무엇이 달라지는지” 출격 전에 한 줄로 설명한다.
- 위협 5 이후에만 엔드리스를 선택지로 둔다.

### 8.4 결과 화면과 재출격

- 시작 함체와 최종 함체 비교
- 완성한 무기·공명·Apex
- 피해 기여도와 보스 TTK
- 발견한 적·기억·설계도
- 다음에 열릴 설계도의 실루엣과 조건
- `같은 빌드 재도전`, `새 조합`, `격납고` 세 경로

### 8.5 짧은 일일 실험

- 10~12분의 선택형 실험 모드를 둘 수 있다.
- 연속 접속 보상이나 미접속 벌점은 두지 않는다.
- 매일 정해진 시작 무기·변형 규칙으로 새 조합을 맛보게 한다.
- 본 캠페인보다 강한 영구 보상을 강제하지 않는다.

## 9. Gate 4 — 스토리·오디오·연출·접근성

### 9.1 세계관과 지역

- 플레이어 기함: NF-0 LUMEN
- 항법 AI: ECHO-7
- 성장 동력: NEON CORE
- 적 집단: THE CHORUS
- B22 네온 아비터: 포획된 최초의 NEON 함선이자 미라 베일의 원형
- B7 하이브 퀸: NEON CORE를 왕관 조각으로 인식

현재 코드와 배경 자산의 지역명은 런타임 ID로 유지한다. 기존 서사 기획의 이름은 대사·기억 장면에서 쓰는 장 제목 또는 부제로 연결한다. 같은 섹터를 두 개의 서로 다른 장소처럼 구현하지 않는다.

| 섹터 | 런타임·배경 지역명 | 서사 장 제목·부제 | 보스 |
|---:|---|---|---|
| 1 | COLD WAKE | Cold Wake | B8 |
| 2 | PRISM GRAVE | Grave Halo | B9 |
| 3 | FURNACE LINE | Ion Cathedral | B10 |
| 4 | BROKEN ARMADA | Crimson Forge | B11 |
| 5 | CHOIR VEIL | Arbiter Gate | B22 |
| 6 | CROWN CORE → CROWN NEST | Crown Nest | B7 |

표시 원칙:

- HUD와 지역 전환 카드에는 런타임·배경 지역명을 우선 표시한다.
- 보스 기억과 이야기 패널에는 서사 장 제목을 부제로 함께 표시할 수 있다.
- 내부 키는 기존 `creative-direction.js`의 지역 ID를 유지해 세이브·테스트 회귀를 막는다.

### 9.2 이야기 전달 규칙

- 인트로: 18~22초, 5패널.
- 전투 중 메시지: 1~2문장, 2.5~4초.
- 같은 메시지 반복 간격: 최소 40초.
- 강화 선택, 고밀도 탄막, 보스 핵심 패턴 중 대사 재생 금지.
- 각 보스 전에는 짧은 등장 문장, 처치 뒤에는 핵심 기억 1개.
- 엔딩은 시작 기함→타이탄→왕관 핵의 의미를 연결한다.
- 반복 출격에서는 이미 본 긴 장면을 축약하거나 건너뛸 수 있다.

### 9.3 이야기·UI 이미지 자산 계약

최종 목표 수량:

| 구분 | 수량 | 사용처 |
|---|---:|---|
| 인트로 패널 | 5 | 18~22초 도입 |
| 보스 기억 | 6 | 각 캠페인 보스 처치 뒤 |
| 엔딩 패널 | 3 | LUMEN·타이탄·왕관 핵 결말 |
| 타이틀 | 1 | 시작 화면 |
| 격납고 | 1 | 메타 성장 화면 |
| 인물·AI 초상 | 최대 11 | ECHO-7, 미라 베일, 기록 인물과 적 신호 |

- 이미지 제작은 Codex가 담당한다. Claude는 임의의 새 이미지를 생성하거나 승인된 이미지를 재디자인하지 않는다.
- Gate 4 구현 전에 Codex가 제공한 파일과 자산 매니페스트를 확인한다.
- 자산이 아직 없으면 레이아웃·로더·폴백까지만 만들고 임시 이미지를 최종 자산인 것처럼 커밋하지 않는다.
- 인간 함선은 백색 세라믹 장갑, 금색 구조선, 절제된 청록 신호를 사용한다.
- CHORUS는 자주·적색 생체 발광, 비대칭 외골격, 호흡하듯 수축하는 움직임을 사용한다.
- 두 진영은 색상뿐 아니라 재질·실루엣·움직임에서도 구분한다.

### 9.4 적응형 음악

기본 구조:

- BASE: 지역 분위기
- DRIVE: 편대 성장과 이동 에너지
- THREAT: 정예·보스 압박
- STINGER: 공명, 함체 승급, BREAK, 보스 처치

필수 전환:

- 지역 전환은 0.8~1.2초 크로스페이드.
- 무기 공명과 함체 승급은 별도 짧은 스팅어.
- B22 BREAK와 B7 심장 노출은 서로 다른 식별 사운드.
- 사운드가 꺼져도 화면 정보만으로 패턴과 강화 등급을 알 수 있어야 한다.
- 외부 라이선스가 불명확한 음원을 저장소에 넣지 않는다. 원본·생성·허가 자산만 사용하고 출처를 매니페스트에 기록한다.

최종 음악 패키지는 지역·보스·메뉴를 포함해 약 12개 트랙 또는 이에 준하는 적응형 stem 세트를 목표로 한다. 숫자를 맞추기 위해 짧은 반복 파일을 억지로 늘리지 말고, 각 지역의 BASE·DRIVE·THREAT 전환과 핵심 STINGER가 실제로 구분되는지를 우선한다.

### 9.5 업그레이드 연출 등급

| 등급 | 사건 | 연출 |
|---|---|---|
| U1 | 일반 수치 보조 | 짧은 포대 플래시와 HUD 펄스 |
| U2 | 행동 변화 카드 | 발사체 형태 전환과 짧은 색 파동 |
| U3 | 무기 진화 | 마운트 재조립, 0.8~1.2초 |
| U4 | 공명·지휘 프레임 | 두 무기 에너지 연결, 전용 스팅어 |
| U5 | H4~H5·Apex | 함체 확장, 엔진 점화, 화면 전체 충격파 |

상위 연출은 하위 연출보다 지속 시간·화면 범위·사운드·카메라 반응 중 최소 세 요소가 커야 한다.

### 9.6 접근성과 성능

- 감소된 모션 모드
- 화면 흔들림 강도 설정
- 광과민성 완화 모드
- 자막과 중요 사운드의 시각 표시
- 색상만이 아니라 모양·위치·소리로 위험 구분
- 390×844와 1280×720 모두 지원
- 현재·다음 지역 자산만 우선 로드
- 저사양에서 입자 수를 줄여도 공격 판정과 정보는 유지
- 프레임 저하 시 게임 시간이 느려져 난이도가 달라지지 않게 dt 상한과 생성량을 검증

## 10. 세이브·데이터 마이그레이션

### 10.1 원칙

- 기존 사용자의 코인, 최고 섹터, 캠페인 클리어, 엔드리스 기록, 사운드 설정을 보존한다.
- 새 필드가 없는 구 저장은 안전한 기본값으로 채운다.
- 저장 버전을 올리고 마이그레이션을 멱등적으로 만든다.
- 같은 저장을 여러 번 불러도 무기 슬롯이나 설계도가 중복 생성되지 않는다.

### 10.2 새 필드 예시

```js
{
  saveVersion,
  unlocks: {
    startingWeapons: [],
    commandFrames: [],
    fleetSystems: [],
    resonanceVariants: []
  },
  blueprints: {},
  threatLevel,
  discoveredEnemies: [],
  bossMemories: [],
  runHistorySummary: []
}
```

런 도중 저장을 지원한다면 `weaponSlots`, `hullIntegrity`, `cruiserHp`, `emergencyRebuilds`, 런 디렉터 사건 상태와 seed까지 함께 저장한다.

## 11. 구현 구조 원칙

- `main.js`에 모든 새 로직을 몰아넣지 않는다.
- 시간 진행, 무기 슬롯, 공명, 내구도, 메타 해금, 통계를 각각 테스트 가능한 순수 로직으로 분리한다.
- 렌더·게임 규칙·저장 데이터가 서로의 내부 구조를 직접 수정하지 않게 공개 함수로 연결한다.
- 공격마다 `sourceWeaponId`, `slot`, `resonanceId`, `attackId`를 추적할 수 있게 한다.
- 밸런스 값은 `balance.js` 또는 역할별 데이터 파일에 모은다.
- UI 문구에 내부 ID를 노출하지 않는다.
- 개발용 치트·테스트 모드는 명시적인 쿼리 또는 dev 페이지에서만 켜진다.

권장 신규 모듈 책임:

- `run-director`: 시간 기반 사건과 8분/25분 진행
- `weapon-loadout`: 슬롯·장착·레벨·진화
- `resonances`: 조합 판정과 발동
- `survivability`: 기함 내구도·순양함 HP·보호막
- `run-metrics`: 사건·피해·TTK 기록
- `meta-progression`: Gate 3 설계도·위협 단계

실제 파일명은 현재 프로젝트 관례에 맞춰 조정해도 되지만 책임 분리는 유지한다.

## 12. 금지 사항

1. 승인된 기함 H0~H5와 무기 마운트·발사체 이미지를 다시 생성하거나 다른 이미지로 교체하지 않는다.
