## 3. 사람 플레이 경로 검증

### 3.1 정직한 시작 상태와 입력 보존

`window.__nfCoreLoop.startPlay()` 직후 실제 런타임 값:

```text
mode=play, auto=false, awaitingStart=true
tier=0, hull=100/100, count=8, banked=0
main=vulcan(선택 전 임시값), wing=null
```

시작 무기 카드를 선택한 뒤 `targetX=80`을 입력하고 120프레임을 진행했다. 함선 x는 `240 → 80.0000005`, 입력 목표는 `80`으로 유지됐다. 플레이 모드에서 자동 운전이 사람 입력을 덮지 않는 것을 실제 런타임으로 확인했다.

### 3.2 실제 선택 UI

390×844 인앱 브라우저에서 다음 순서로 직접 클릭했다.

1. 0:00 `시작 무기 선택` — 발칸/레이저/유도 미사일 3장
2. 0:30 `행동 변화 강화` — 선택 무기 Lv2
3. 1:10 `행동 변화 강화` — 선택 무기 Lv3
4. 1:15 `두 번째 무기 획득` — 기존 무기를 유지한 채 레이저/유도 미사일을 보조 하드포인트에 장착

선택 중 게임 시간이 멈추고, 선택 뒤 다시 진행했다. 두 번째 무기 선택 직후 실제 HUD와 결과 화면에 주무기+보조무기 조합이 표시됐다.

### 3.3 8분 플레이 모드 완주 가능성

사람 플레이 모드(`auto=false`)에 외부 회피 입력만 공급하고 선택 카드를 실제 DOM 클릭으로 처리한 8분 재생 결과:

```text
durationSec=480, gameOverReason=clear
B22 TTK=46.0초, resonanceShare=25.4%
시작 H0·내구도100 → 종료 H1·내구도111/122
주무기 vulcan + 보조무기 laser
```

내부 자동 운전을 켠 측정 모드가 아니라 플레이 모드의 입력 채널로 완주했다. 따라서 H0·내구도 100에서 선택→성장→B22→결과까지 연결되는 수직 슬라이스가 실제로 존재한다.

보수적으로 기록할 관찰도 있다. 390×844에서 단순 좌우 왕복만 한 수동 조작은 1:15 두 번째 무기 선택 직후 내구도 0으로 종료됐다. 결과 화면은 실제 상태(스카우트→스카우트, 발칸+레이저, 내구도 0/100, 긴급 재건 1회)를 정확히 표시했다. 이는 기능 결함 판정이 아니라 요청서가 Gate 2로 이월한 **실제 사용자 생존 곡선 조정이 여전히 필요하다**는 근거다.

## 4. 자동 측정 재생

동일 페이지의 `window.__nfCoreLoop.run(buildId, 700)`을 각 빌드별로 새로 실행한 결과다. 제출서의 고정 숫자와 완전히 같지는 않았으나 모든 값이 계약 범위 안에 재현됐다.
