## 1. 수정한 결함 3건

### 결함 1 — 순양함 합체 후 드론 0기 무적 회귀 (최우선)

**원인.** 직전 커밋 `504ed64`가 "합체로 count=0이 된 것을 전멸로 오판해 강등되는 버그"를 막으려고
`before > 0 && count === 0` 조건을 썼는데, 합체로 count가 **이미 0**이 되면 그다음 호출부터는
`before`가 0이라 이 조건이 영영 참이 될 수 없었다. 결과적으로 어떤 피해를 받아도 안전망이
발동하지 않는 **사실상 무적** 상태가 됐다.

서로 반대 방향의 두 계약을 한 조건으로 처리하려던 것이 근본 원인이다.

- (A) **성장 소모**로 0 → 강등·사망 금지 (드론 130기 → 순양함 1척 합체 시 잔여 0)
- (B) **실제 피해**로 0 → 안전망 정확히 1회 작동 (합체로 이미 0이던 상태 포함)

**수정.** 판정 기준을 '직전 값'이 아니라 **'이번 델타의 부호'**로 바꾸고, `checkEvolution()` **이전 시점**에 평가한다.

```js
const depletedByLoss = n < 0 && this.count === 0;   // 손실로 0이 됐는가 (합체 이전 시점)
...
if (depletedByLoss) { this.onDronesDepleted(world); return; }
this.checkEvolution(world);
```

- `n > 0` / `n = 0`이 합체로 0이 되어도 미발동 → (A) 유지
- count가 이미 0이어도 `n < 0`이면 발동 → (B) 회복
- 음수 델타가 합체를 유발해 0이 된 경우(200 → 피해 → 130 → 합체 → 0)는 판정이 합체 이전이라 미발동

### 결함 2 — 캠페인 기록과 무한 원정 기록 완전 분리

**원인.** `startSector()`와 `endExpedition()`이 모드 구분 없이 공용 `stage`를 갱신해,
캠페인을 섹터 6에서 끝낸 사용자가 무한 원정을 시작하는 순간 캠페인 기록이 섹터 7 이상으로 오염됐다.

**수정.** 순수 함수 `logic.progressPatch(mode, sector, data)`를 추출하고 저장 지점 3곳을 전부 경유시켰다.

- `campaign` → `{ stage }`만, `endless` → `{ endlessBest }`만 반환 (**엔드리스 패치에는 `stage` 키 자체가 없음**)
- 기존 기록보다 크지 않으면 빈 패치(저장 변경 없음)
- 적용 지점: `startSector` / `endExpedition` / `winCampaign` — `main.js`에 `stage`를 직접 쓰는 `save.set`은 남아 있지 않다

### 결함 3 — Pretendard 로딩 오류 제거

**원인.** 기존 CSS의 jsDelivr 주소가 **HTTP 403**. 저장소 구조 변경으로 실제 경로가
`packages/pretendard/dist/...`로 이동해 옛 `gh/.../dist/...` 경로가 죽었다. 페이지를 열 때마다
콘솔 NetworkError가 남고 글꼴 적용도 보장되지 않았다.

**수정.** 공식 배포본을 저장소에 자체 포함(self-host)하고 CSS를 상대 경로로 전환.

- **출처**: npm 공식 패키지 `pretendard@1.3.9` → `dist/web/static/woff2` (jsDelivr 경유, 버전 고정)
- **라이선스**: SIL Open Font License 1.1 (Copyright 2021 Kil Hyung-jin) — 원문을 `assets/fonts/LICENSE-Pretendard.txt`에 동봉
- 실제 사용하는 **Regular 400 / Bold 700 두 굵기만** 포함 (전체 9굵기 패키지 미포함)
- woff2 매직바이트(`wOF2`) 검증 완료

## 5. 보스 난이도 측정 (§6) — 측정만 수행, 수치 미변경

현재 설정 `globalMult 1.38 · bossHpMult 2.2 · bossRateMult 1.8` (이번 커밋에서 **변경하지 않음**).

조건: 격납고 무강화, 각 섹터 도달 시점의 대표 편대로 구성, 트랙 구간은 측정 대상에서 제외(보스전만 계측).

| 섹터 | 보스 | 기수 | 시작 함대 화력 | 보스 총 HP | HP/화력 | 실측 DPS | 처치시간(완벽 회피) | 회피 없이 방치 시 |
|---|---|---|---|---|---|---|---|---|
| 1 | 리퍼 로드 (B8) | 1 | 200 | 16,364 | **82×** | 247 | **66초** | 10.5초 만에 전멸 (피격 7회, 드론 150→0) |
| 3 | 옵시디언 클로 (B10) | 3 | 1,500 | 420,411 | **280×** | 1,778 | **237초 (약 4분)** | 11.0초 만에 전멸 (피격 29회, 드론 200→0) |
| 6 | 하이브 퀸 (B7) | 3 | 5,520 | 2,066,424 | **374×** | 13,495 | **153초 (약 2.5분)** | 12.9초 만에 전멸 (피격 28회, 드론 22→0) |

### 해석

1. **"멈춰서 격파" 문제는 해결됐다.** 세 섹터 모두 회피하지 않으면 10~13초 안에 전멸한다.
   `bossRateMult 1.8`이 실제로 걸린 결과다(직전까지 보스는 `scaleEnemy`를 안 거쳐 발사 버프를 못 받고 있었다).
2. **대신 보스전이 지나치게 길어졌다.** 완벽 회피 기준 처치시간이 66초 / 237초 / 153초다.
   특히 섹터 3은 **4분**으로, 일반 노드(수십 초)와의 난이도·시간 계단이 매우 크다.
3. **곡선이 비단조다.** 섹터 3(237초)이 섹터 6(153초)보다 오래 걸린다. 고화력 구간에서는
   `hpPerPowerCap`이 걸려 HP 증가가 멈추는 반면 플레이어 DPS는 계속 오르기 때문이다.
4. **무강화 섹터 1 공략 가능성**: 가능하지만 **66초간 한 대도 안 맞아야** 하는 수준이다(피격 7회면 전멸).
   신규 플레이어에게는 사실상 벽에 가깝다.

2. **이미 오염된 저장 데이터** — 이번 수정은 앞으로의 오염을 막지만, 수정 이전에 무한 원정을 돌려
   `stage`가 7 이상으로 기록된 사용자의 값을 되돌리지는 않는다(기존 기록 보존 원칙 §4.2-8을 우선).
   필요하면 캠페인 `stage`를 6으로 정리하는 1회성 마이그레이션을 별도로 논의한다.
3. **글꼴 용량** — Regular+Bold 합계 약 1.5 MB. 기존 에셋 총량(38 MB) 대비 미미하나,
   더 줄이려면 공식 subset 판본(각 약 267 KB)으로 교체 가능하다. 다만 subset은 수록 글리프가 제한되어
   한글 깨짐 위험이 있어 이번에는 full static을 택했다.
